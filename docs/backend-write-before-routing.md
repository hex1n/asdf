# 写前路由(后端需求)

> 用途:后端日常需求**开写之前**的决策清单,决定动手前先用哪个 skill(`/deep-research`、
> first-principles 选型、`/grill-me`、`/grill-with-docs`)。
>
> 放置位置:这是工作流设计草稿。真正落地时,把"路由"与"复利"两节复制进**你公司工作仓的
> `CLAUDE.md`**,让 agent 自动读取并在写码前主动提醒。它引用了 Claude Code 专属的 `/grill-*`
> 命令,**不放进本仓 `skills/`**(违反 asdf 的可移植性约定),故留在 `docs/`。
>
> 设计依据:见同会话推导——按"最不可逆的未知"路由,而非按"代码熟不熟"分层。

## 路由

原则:按**最不可逆的未知**选工具。
铁律:每加一个工具,要能说出它对应哪个**被命名的未知**;说不出就别加。
通则覆盖长尾:凡能说出一个承重且未核实的未知(如"下游 API 真会返回 X 吗"、
"这设计扛得住峰值吗"),即触发 ①——外部依赖、容量/性能等不另起行。

**给 agent:实现任何后端改动前,先用此表分类主导未知,并在动手前提议对应步骤。**

| 问自己 | 是 → 动作 | 后端触发例 |
|---|---|---|
| ① 现状/为什么我没把握? | 先 `/deep-research` | 没摸过的链路、历史逻辑不明、不确定有无现成机制可复用、依赖的外部系统行为未知 |
| ② 欠规格(只给解法没给目标)或有多个真机制? | 第一性原理重构根问题再选型 | 自建 vs 复用、同步 vs 异步、加表 vs 改表 |
| ③ **改动不可逆?**(改表/迁移/回填/契约/灰度回滚) | **设计必须含 迁移+回滚+兼容 方案**;优先级最高,不可跳 | online migration、双写、破坏性 schema 变更、跨服务契约 |
| ④ 决策分支没收敛?(幂等/事务边界/部分失败/超时重试) | `/grill-me`(脊柱步;仅"真·局部可逆改"可跳) | 并发幂等、事务边界、部分失败回滚、重试语义 |
| ⑤ 碰领域语义或长期取舍? | ④ 叠加 docs → `/grill-with-docs` 顺带产 ADR+术语 | 新领域概念、状态机、跨团队契约 |
| ⑥ 收口 | grill 时**定下验收标准**;实现后按其验证(必要时 `tdd`/`diagnosing-bugs`) | 验收点写不出 = 需求还没想清,回 ②/④ |
| 全 no | 直接写 | 局部小改、加字段、改文案、熟悉路径 |

## 护栏 & 校准

- 异质 ≠ 每单跑满;只有 ④ 是脊柱,其余靠未知触发。
- `/grill-with-docs` 用完盯住别越界直接改代码(上游 [Issue #134](https://github.com/mattpocock/skills/issues/134))。
- **校准 tally**:每单合并后在 `docs/rework-log.md` 记一行。多来自①→`/deep-research` 默认调高;
  多来自④→grill 介入太晚;多来自③→迁移 gate 没守住。

## 复利(系统级)

每单的产出要去**降低未来单的 ①**,否则每单都在重交学费:

- `/grill-with-docs` 产的 ADR/术语、`rework-log` 记的教训,必须汇进**团队可检索知识库**
  (同 ① 的搜索入口)。
- 目标:让"现状未知"的发生率随时间下降——闭的是**系统**的环,不只是单内的环。

## 校准记录怎么用

`docs/rework-log.md`(模板见同目录)。冻结此设计,**先跑 5 单再改**。下一次该改这张表的信号,
是 `rework-log` 显示某类未知反复漏网,而不是又一轮纸面 review。
