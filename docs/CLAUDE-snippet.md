> 这是去掉草稿说明的可粘贴版;完整设计依据见 [backend-write-before-routing.md](backend-write-before-routing.md)。
>
> 粘贴注意:
> - `②` 行无 slash 命令——first-principles-planner 是 asdf 源 skill,工作仓运行时若没装,
>   该步靠片段内"给 agent"祈使触发;要 `/` 触发需先把该 skill 装进 Claude Code。
> - 同时把 `docs/rework-log.md` 模板搬进工作仓,否则校准那行没有落点。

```markdown
## 写前路由(后端需求)

原则:按**最不可逆的未知**选工具,而非按"代码熟不熟"分层。
铁律:每加一个工具,要能说出它对应哪个**被命名的未知**;说不出就别加。
通则覆盖长尾:凡能说出一个承重且未核实的未知(如"下游 API 真会返回 X 吗"、
"这设计扛得住峰值吗"),即触发 ①——外部依赖、容量/性能等不另起行。

**给 agent:实现任何后端改动前,先用此表分类主导未知,并在动手前提议对应步骤。**

| 问自己 | 是 → 动作 | 后端触发例 |
|---|---|---|
| ① 现状/为什么我没把握? | 先 `/deep-research` | 没摸过的链路、历史逻辑不明、不确定有无现成机制可复用、依赖的外部系统行为未知 |
| ② 欠规格(只给解法没给目标)或有多个真机制? | 第一性原理重构根问题再选型 | 自建 vs 复用、同步 vs 异步、加表 vs 改表 |
| ③ **改动不可逆?**(改表/迁移/回填/契约/灰度回滚) | **设计必须含 迁移+回滚+兼容 方案**;优先级最高,不可跳 | online migration、双写、破坏性 schema 变更、跨服务契约 |
| ④ 决策分支没收敛?(幂等/事务边界/部分失败/超时重试) | `/grill-me`(脊柱步;仅"真·局部可逆改"可跳) | 并发幂等、事务边界、部分失败回滚、重试语义 |
| ⑤ 碰领域语义或长期取舍? | ④ 叠加 docs → `/grill-with-docs` 顺带产 ADR+术语 | 新领域概念、状态机、跨团队契约 |
| ⑥ 收口 | grill 时**定下验收标准**;实现后按其验证(必要时 `tdd`/`diagnosing-bugs`) | 验收点写不出 = 需求还没想清,回 ②/④ |
| 全 no | 直接写 | 局部小改、加字段、改文案、熟悉路径 |

### 护栏 & 校准
- 异质 ≠ 每单跑满;只有 ④ 是脊柱,其余靠未知触发。
- `/grill-with-docs` 用完盯住别越界直接改代码。
- 校准:每单合并后在 `docs/rework-log.md` 记一行。多来自①→`/deep-research` 默认调高;
  多来自④→grill 介入太晚;多来自③→迁移 gate 没守住。

### 复利(系统级)
每单产出要降低未来单的 ①:`/grill-with-docs` 的 ADR/术语、`rework-log` 的教训,
必须汇进**团队可检索知识库**(同 ① 的搜索入口),让"现状未知"发生率随时间下降。
```
